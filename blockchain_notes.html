3.1

Introduction to Solidity

What is Solidity?

Solidity is the primary programming language for writing smart contracts on the Ethereum blockchain. It is a contract-oriented, statically-typed, high-level language that compiles down to EVM (Ethereum Virtual Machine) bytecode.

  • Syntax resembles JavaScript, Python, and C++
  • Designed specifically to run on the Ethereum Virtual Machine (EVM)
  • The EVM is hosted on Ethereum Nodes connected to the blockchain
  • Smart contracts in Solidity store all programming logic that transacts with the blockchain
What is Ethereum?
  • A decentralized open-source platform based on blockchain
  • Runs smart contracts — applications that execute exactly as programmed with no possibility of fraud, downtime, or third-party interference
  • Platform for nearly 2,60,000 different cryptocurrencies
  • Miners earn Ether as rewards for securing the blockchain
What is the EVM?
  • EVM = Ethereum Virtual Machine
  • A runtime environment for executing smart contracts
  • Provides security and executes untrusted code via a public node network
  • Prevents DoS attacks; ensures program isolation (contracts can't access each other's state)
  • Ensures communication with no potential interference
Why Use Solidity for Smart Contracts?
  • Security Focus: Prioritizes safety for handling valuable blockchain assets
  • Blockchain Integration: Tailored for Ethereum's infrastructure
  • Decentralized Execution: Deterministic execution across all nodes
  • Smart Contract Features: Built-in logic for contracts, events, access control
💡
Smart Contracts — high-level program codes compiled to EVM bytecode and deployed to the Ethereum blockchain. They allow credible, trackable, and irreversible transactions without any third-party.

Constructors

A constructor is an optional function executed exactly once — at contract deployment. It initializes the contract's state.

constructor() <Access Modifier> { // initialization code here }
  • Customization: Set initial state by passing values during deployment
  • Efficiency: Runs only once — more efficient than regular function calls
  • Security: Proper init avoids dangerous uninitialized states
  • Code Clarity: Makes intent clear
  • Can be public or internal; internal makes contract abstract

Control Flow & Error Handling

Control Flow Statements
  • Conditionals: if, else if, else, switch
  • Loops: for, while, do while
  • Jump: break, continue, return
  • Error Handling: assert, require, revert, custom errors
assert()
  • Purpose: Internal invariant checks (conditions that should NEVER be false)
  • Gas: Consumes ALL remaining gas (no refund)
  • Use when: Bug detection only
assert(_value > 100);
require()
  • Purpose: Input validation & preconditions
  • Gas: Refunds unused gas ✅
  • Use when: Normal validation that can fail
require(_val > 100, "Too small");
revert()
  • Purpose: Flexible conditional errors
  • Gas: Refunds unused gas ✅
  • Use when: Complex conditions, try/catch
if(_v <= 100) revert("Too small");

Arrays, Strings, Enums, Structs & Mappings

Arrays
  • Fixed-size: uint[3] fixedArray; — size defined at declaration
  • Dynamic: uint[] dynamicArray; — can grow/shrink

Strings
  • Sequence of characters in UTF-8 encoding
  • Can contain any Unicode character
  • No fixed size — grows dynamically
  • string public name = "Hello";
  • Strings are reference types — stored in memory or storage
Enums

User-defined types for a fixed set of named constants. Replaces "magic numbers" with readable names.

  • First value = 0, second = 1, and so on (stored as integers)
  • Useful for representing states (Active, Inactive, etc.)
enum OrderStatus { Placed, Shipped, Delivered, Cancelled } OrderStatus public status;
Structs

Custom data types that group related variables of different types into one unit.

  • Combine multiple fields into a single type
  • Useful for user profiles, records, products
  • Define once, reuse multiple times
struct Product { uint id; string name; uint price; }
Mappings

Data structures storing key-value pairs — like hash tables. Very efficient for storage and retrieval.

  • Key-Value Storage: Each unique key maps to exactly one value
  • Default Values: Unset keys return the default for the value type (0, false, "")
  • Efficient Lookup: O(1) constant-time retrieval
  • Cannot Iterate: No looping through all entries directly
mapping(address => uint) public balances; balances[msg.sender] += msg.value; // add to mapping
3.9

Payable Functions & Fallback

Payable Functions

Functions marked payable allow a smart contract to accept Ether. Without this keyword, sending Ether to a function causes the transaction to revert.

function deposit() payable external { // Ether is automatically stored in contract's balance balances[msg.sender] += msg.value; // record who sent what }
💡
If a function is NOT payable and someone sends Ether to it, the transaction will revert and funds are returned (minus gas cost).
receive() Function

A special built-in function that gets called when the contract receives Ether with empty calldata. Allows contracts to receive Ether like a regular wallet.

receive() external payable { // Called when msg.data is empty AND msg.value > 0 // No calldata needed }
Fallback Function

A unique function automatically executed when a contract receives a transaction that doesn't match any function signature, OR when msg.data is not empty.

fallback() external payable { // Handles unknown function calls }
📋
Fallback Function Rules:
  • Declared with fallback() keyword — no arguments
  • Must be marked external
  • Cannot return anything
  • Only one fallback per contract
  • Limited to 2300 gas when called via transfer() or send()
  • If not marked payable, throws exception when receiving plain Ether
Ether Routing Logic
Ether sent to contract | msg.data is empty? / \ YES NO | | receive() exists? fallback() / \ YES NO | | receive() fallback()
Uses of Fallback Functions
  • Handling Unexpected Transactions: Process unknown calls with custom error handling
  • Custom Payment Processing: Accept specific tokens or do validation before altering state
  • Gas Optimization: Accept compressed data formats to save gas
3.10

Events & Function Overloading

Solidity Events

Events are the EVM's logging mechanism — they write information to a data structure on the blockchain that lives outside of smart contracts' storage variables. Used to notify external apps (dApps) of state changes.

// Define an event event NewTransaction(address indexed sender, uint amount); // Emit the event inside a function function sendFunds() public payable { emit NewTransaction(msg.sender, msg.value); }
  • Defined using the event keyword
  • Triggered (emitted) using emit EventName(...)
  • dApps listen to events using libraries like ethers.js via the Ethereum JSON-RPC API
  • indexed parameters allow efficient filtering and searching of event logs
Function Overloading

Solidity allows multiple functions with the same name but different parameter types or counts. The compiler resolves which version to call based on the arguments passed.

function add(uint a, uint b) public pure returns (uint) { return a + b; } function add(uint a, uint b, uint c) public pure returns (uint) { return a + b + c; }
  • Compiler searches for a function with matching name AND parameter types
  • If no matching function is found → compilation error
  • Cannot overload based on return type alone

Withdrawal Patterns & Access Control

Withdrawal Patterns

Strategies for handling fund transfers securely, protecting against reentrancy attacks.

❌
Simple / Vulnerable Pattern
Transfer funds to user BEFORE updating balance. Attacker can recursively call withdraw before balance is updated → Reentrancy Attack
✅
Pull Over Push (Safe)
Update balance FIRST, then transfer. User must actively "pull" funds. State updated before external transfer → Reentrancy-safe
// Safe: Pull Over Push Pattern mapping(address => uint) public balances; function withdraw(uint amount) public { require(balances[msg.sender] >= amount, "Insufficient balance"); balances[msg.sender] -= amount; // UPDATE STATE FIRST payable(msg.sender).transfer(amount); // THEN transfer }
Access Control

Access control mechanisms regulate who can interact with specific functions or data. They enforce security policies and ensure only authorized users execute certain functions.

Modifiers for Access Control

Modifiers are special functions that augment the behavior of other functions. Applied using the modifier keyword.

🔐
onlyBy — Restricts function to a specific address only

require(msg.sender == _account);
⏰
onlyAfter — Function can only be called after a specific time

require(block.timestamp >= _time);
💰
costs — Caller must provide a minimum value

require(msg.value >= _value);
modifier onlyBy(address _account) { require(msg.sender == _account, "Only specified address can call"); _; // continue with the function body } function changeOwner(address _newOwner) public onlyBy(owner) { owner = _newOwner; }

4.2

Operation Mechanism Risks

51% Attack

A 51% attack occurs when a single entity (or group) gains control of more than 50% of the network's total hashing power, allowing them to manipulate the transaction history.

⚠️
What the Attacker Can Do:
  • Double Spending: Reverse transactions on the honest chain and reuse tokens on the malicious chain
  • Block New Transactions: Prevent transactions from being confirmed, effectively freezing the network
  • Disrupt Confidence: Shatter trust in the network's immutability
Mitigation Strategies
  • Higher Network Hashrate: Makes 51% computationally and economically unviable
  • Checkpointing / Finality Rules: Blocks after a certain depth cannot be changed
  • Switch to Proof of Stake (PoS): Reduces power centralization (e.g., Ethereum 2.0)
  • Monitor Mining Pools: Enforce policies preventing any pool from exceeding 50%
Private Key Security

In blockchain, ownership of digital assets is determined entirely by who controls the private key. Unlike traditional login systems, there is no username/password — just cryptographic keys.

🔑
Attack Scenario: A hacker steals Alice's private key → Impersonates Alice → Signs fraudulent messages → The network accepts them (public key still matches) → Funds stolen, irreversibly.
Best Practices
  • Hardware Wallets: Store keys offline, away from hackers
  • Multi-Sig Wallets: Require multiple keys to approve a transaction
  • Never Plaintext Storage: Avoid screenshots, cloud storage, or emails
  • Strong Passwords + 2FA: Even for wallets and exchange accounts
  • Encrypted Backups: USB, cold storage, multiple locations
Criminal Activities

Blockchain enables fast, borderless, pseudonymous transactions — ideal for innovation but exploitable for illegal activities.

💻
Ransomware
Encrypted files held for crypto ransom
🌑
Dark Web Markets
Illicit goods traded with crypto
🧹

Clean funds by mixing crypto
Countermeasures
  • Chain Analytics Tools (Chainalysis, CipherTrace) — track suspicious fund flows
  • KYC/AML Compliance on exchanges — verify user identity
  • Regulation of Privacy Coins — legal restrictions on Monero, Zcash, mixers
Double Spending

Spending the same cryptocurrency more than once. Unlike physical cash, digital currencies can be copied unless properly validated.

1
Fake Trust

Attacker sends TXₐ to a vendor (honest chain) and simultaneously creates TXᵦ sending same coins to their own address

2
Mining the Real Transaction

Attacker ensures TXᵦ gets mined (via hash power or colluding pool). TXₐ to the vendor becomes invalid as blockchain accepts longest chain

3
Exploiting the Race

Attacker receives goods from vendor (who thought TXₐ was legitimate). Only TXᵦ is on the blockchain. Attacker keeps BOTH the goods AND the money.

✅
Countermeasures: Wait for multiple confirmations • Use real-time payment gateways • Employ double-spending detection tools • Avoid zero-confirmation payments for valuable goods
Transaction Privacy Leakage

Most blockchains are pseudonymous, not anonymous. Addresses aren't directly tied to identities, but all transactions are public and traceable.

  • Privacy leakage = observer deduces identity/behavior from transaction patterns, linkages, or metadata
  • Implications: De-anonymization, government surveillance, exposure of business strategies, risk to activists/journalists
  • Mitigations: Privacy coins (Monero, Zcash), avoid address reuse, coin mixers, fresh addresses per transaction, use Tor/VPN

Blockchain Network Attacks

📊
Real-World Impact: 2022: ~$3.7B stolen from crypto platforms | 2023: $1.7B (−54.3%) | 2024: $2.2B (+21%). Number of incidents continues rising each year.
DDoS (Distributed Denial of Service)

Targets blockchain availability by overwhelming nodes, RPC endpoints, or the mempool with massive traffic. Doesn't compromise integrity but can cripple usability and inflate gas fees.

TargetAttack TypeEffect
Full NodesNetwork FloodingSaturate bandwidth or memory
Smart ContractsLogic Bombs / LoopingInfinite loops, expensive operations
Mining/Validator PoolsConsensus DisruptionDelay validation
MempoolTransaction SpammingClog the pending tx pool
OraclesData Query FloodsManipulate or break oracle data
Mitigation
  • Gas Fees — Economic deterrent; spamming becomes expensive
  • Mempool Filtering — Reject low-fee/duplicate tx, use Replace-by-Fee
  • Rate Limiting — Control requests from a single source
  • Load Balancing — Distribute traffic across multiple nodes
  • CDN — Offload traffic from primary infrastructure
  • Cloud-Based DDoS Protection — Absorb large-scale attacks
Transaction Malleability Attack

The ability to alter a transaction's hash (TXID) without changing its effect. An attacker tweaks a transaction slightly, giving it a different TXID, causing confusion in systems that track by TXID.

  • Problem: Wallets/exchanges tracking TXID may think a tx failed and resend → double spending
  • Fix: SegWit (Segregated Witness) — separates signature from TXID hash computation
  • Bitcoin adopted SegWit in 2017 to eliminate this class of attacks
Time Jacking

An attacker manipulates the network time perceived by a node by flooding it with Sybil nodes that send incorrect timestamps. The victim's internal clock shifts, causing invalid block acceptance or forks.

GoalImpact
Desynchronize nodeAccept outdated or invalid blocks
Force forksExploit differences in chain view
CensorshipDelay confirmation of valid transactions
Denial of ServiceNode becomes stuck or disconnected
  • Countermeasure: Bitcoin Core sets bounds of ±70 minutes from system clock
  • Rely on system time rather than peer time
  • NTP integration for time verification
Routing Attacks

Attacker manipulates internet traffic routing protocols (BGP, DNS) that blockchain nodes depend on for synchronization.

  • Partitioning Attack: Splits blockchain network into disjoint segments → forks, stale blocks, double-spending risk
  • Delay Attack: Slows traffic between nodes → nodes get outdated info → attacker mining advantage
  • Hijack Attack: Misleads nodes through malicious networks (BGP hijack) → monitor/modify/discard data
  • Mitigations: Multiple ISPs, encrypted comms (VPN/Tor), network monitoring, diversified peers, Route Origin Validation (ROV)
Sybil Attacks

An attacker creates multiple fake identities (nodes) in the network. The victim is surrounded by fake nodes that intercept all their transactions, making them vulnerable to double spending.

  • Difficult to detect and prevent
  • Countermeasures: Increase cost of creating new identity | Require trust for joining network | Determine user power based on reputation
Eclipse Attacks

Attacker controls all peer connections of a victim node, cutting it off from honest nodes. The victim becomes entirely dependent on the attacker's (possibly false) view of the blockchain.

1
Peer Table Flooding

Attacker floods victim's peer table with fake/malicious IPs, crowding out real peers

2
Restart Exploit

Node reloads peers from disk on restart. Attackers stored as preferred peers reconnect automatically

3
Full Isolation

Attacker controls all connections → can send old blocks, hide mempool transactions, feed fake blockchain forks

✅
Mitigation: Diversify connections (no all-from-one-IP) | Authenticated peers / whitelists | Rate-limit new peer additions | Periodic peer rotation
Long-Range Attacks (PoS)

Targets Proof of Stake blockchains by reconstructing an alternative history from a point far back in the chain using old private keys. Unlike 51% attacks, no massive energy is needed — just old validator keys.

1
Obtain Old Keys

Attacker acquires private keys of validators who had high stake in the past

2
Rebuild Chain from Genesis

Creates a valid alternative chain from a distant checkpoint using those old keys

3
Convince New Nodes

New/offline nodes syncing may adopt the malicious chain if it appears valid and follows protocol rules

✅
Mitigations: Checkpointing (ignore chains before trusted checkpoint) | Weak Subjectivity (new nodes get trusted snapshot) | Slashing (penalize double-signing) | Key Expiry & Rotation
4.4

Smart Contract Attacks

Top 5 Smart Contract Attacks
  • Reentrancy Attack
  • Integer Overflow / Underflow
  • Front-Running (Transaction Ordering Dependency)
  • Flash Loan Attacks
  • Unchecked External Calls
1. Reentrancy Attack

Exploits coding vulnerabilities that allow an external contract to re-enter a function before the state is updated. The most famous example is The DAO hack (2016).

⚠️
How it works: Contract sends ETH to external address → external contract's fallback runs → before original contract updates balance, attacker calls withdraw again → drains funds repeatedly
Mitigation
  • Checks-Effects-Interactions pattern: Validate → Update state → External call
  • Reentrancy Guard / Mutex: Modifier preventing function re-entry while already executing
  • Avoid call() unless necessary: Prefer transfer() or send() which limit gas
  • Pull Over Push pattern for fund transfers
2. Integer Overflow/Underflow

In older Solidity (pre-0.8.0), when a calculation exceeds the max value or drops below zero, it wraps around silently.

⬆️
Overflow: uint8 max is 255. If you add 1 → it becomes 0 silently! Attacker can wrap balances around to gain funds.
⬇️
Underflow: uint8 value 0. Subtract 1 → it becomes 255 silently! Attacker can gain a massive balance from nothing.
✅
Fix: Use Solidity ≥ 0.8.0 (built-in overflow checks) | Use SafeMath library for older versions | Always validate arithmetic manually if unsure
3. Front-Running

An attacker observes a pending transaction in the public mempool and submits their own transaction with a higher gas fee, causing it to be mined first.

1
Alice's TX in Mempool

Alice submits a buy order on Uniswap for a token

2
Bob Sees It

Bob (attacker/MEV bot) sees the pending transaction and knows it will push the price up

3
Bob Frontruns

Bob submits the same buy with higher gas → mined first → price rises

4
Profit

Alice's tx mines after at the higher price. Bob immediately sells — profiting at Alice's expense (sandwich attack)

✅
Mitigation: Commit-Reveal scheme (commit hash first, reveal later) | Randomized/delayed execution | Private transaction relays (Flashbots) — submit privately to miners
5. Unchecked External Calls

When a smart contract calls another contract or address, it temporarily gives up control. If the external call fails silently or misbehaves and the result isn't checked, bad things happen.

  • Types: address.call(), address.send(), address.transfer(), externalContract.someFunction()
  • Can't be sure what external code will do
  • Might fail without you knowing (if not checking return value)
  • Might re-enter your contract (reentrancy)
  • Might consume too much gas
✅
Mitigation: Always check return values | Avoid calling unknown contracts | Use pull instead of push | Use limited gas forwarding
Key Defense Best Practices
  • Use latest Solidity version (≥ 0.8.0) for built-in safety
  • Avoid external calls or check them rigorously
  • Follow Checks-Effects-Interactions pattern always
  • Use OpenZeppelin libraries for security-hardened patterns
  • Rigorously test and audit all logic using tools like Slither, MythX, and Foundry

Selfish Mining

What is Selfish Mining?

A deceptive strategy where a miner (or group) solves a block but withholds it from the public blockchain, creating a private fork to gain an unfair advantage over honest miners.

1
Mine Privately

Selfish miner finds a block but withholds it from the network

2
Extend Private Chain

Continues mining on its own secret chain to stay ahead

3
Wait for Public Catch-Up

If public miners find a block, the selfish miner quickly releases their chain

4
Longest Chain Wins

Network accepts selfish miner's chain (if longer), orphaning honest miners' blocks

5
Unfair Reward

Selfish miner earns more than their share of mining power; honest miners waste resources

Why It Works
  • Most blockchains follow the longest-chain rule
  • Honest miners waste resources on blocks that get orphaned
  • Selfish miners get more frequent rewards by manipulating chain visibility
Impact & Mitigation
  • Centralization Risk: Encourages large mining pools
  • Security Degradation: Reduces trust in PoW consensus
  • Lower Network Efficiency: More orphaned blocks, wasted resources
  • Fairness Violation: Honest miners unfairly penalized
  • Modify consensus to penalize block withholding
  • Uncle/orphan rewards (Ethereum) reduce selfish incentives
  • Detect abnormal chain behaviors using statistical/game-theoretic models
  • No known real-world attacks as of 2021

4.7

Blockchain-Based DNS Security

Traditional DNS Problems

DNS (Domain Name System) translates human-readable domain names (like google.com) into IP addresses. Traditional DNS relies on centralized servers managed by companies or governments.

  • DNS Spoofing/Poisoning: Attacker sends fake DNS responses, rerouting users to malicious sites
  • Single Point of Failure: If a DNS server is down or hacked, domains become unreachable
  • Censorship: Governments/ISPs can block or manipulate domain names
  • Lack of Transparency: No public record of DNS entry changes
How Blockchain Fixes DNS

A blockchain-based DNS replaces central registries with a decentralized ledger maintained by multiple nodes where domain ownership is recorded as blockchain transactions.

  • Entries can't be altered without private keys
  • Every change is publicly verifiable on-chain
  • No central body controls access or domain revocation
  • Smart contracts enforce rules (renewals, transfers)
  • Decentralized — no single point of failure
Real-World Examples
🌐
ENS (Ethereum Name Service)
Uses Ethereum smart contracts. Maps .eth domains to wallet addresses or IPFS sites.
🤝
Handshake (HNS)
Decentralized root DNS zone. Competes with ICANN. Auction-based TLD registration.
🔑
Unstoppable Domains
.crypto, .wallet, .x domains. Links to multiple wallets. One-time payment, no renewals.
4.8

Hyperledger Fabric Architecture

Node Types in Hyperledger Fabric
📦
Committing Peer
Maintains ledger and world state. Commits transactions. May hold chaincode (smart contract).
✅
Endorsing Peer
Specialized committing peer. Receives transaction proposals and grants/denies endorsement. Must hold chaincode.
📋
Ordering Node
Approves transaction block inclusion in the ledger. Does NOT hold chaincode or ledger. Communicates with peers.
7-Step Transaction Flow
1
Propose Transaction

Client application submits a transaction proposal to required endorsing peers (as per endorsement policy)

2
Execute Proposed Transaction

Endorsers execute proposals independently — NO ledger update yet. Each captures Read-Write (RW) sets.

3
Proposal Response

RW sets are signed by each endorser and returned to the application asynchronously

4
Order Transaction

Application submits responses to the ordering service. Ordering happens in parallel for all applications.

5
Deliver Transaction

Ordering service packages transactions into blocks and delivers to committing peers

6
Validate Transaction

Every committing peer validates against endorsement policy AND checks RW sets against current world state. Valid tx → applied to state. Invalid tx → recorded but world state NOT updated.

7
Notify Transaction

Applications registered for events are notified of success/failure for each transaction and new block additions

Ordering Service Options
  • SOLO: Single node — for development only
  • Kafka: Crash fault tolerant consensus — minimum 3 nodes, odd number recommended
Channels

Channels provide privacy between different ledgers within the same Fabric network.

  • Ledgers exist in the scope of a channel
  • Can be shared across an entire network or limited to specific participants
  • Chaincode is instantiated on specific channels
  • Peers can participate in multiple channels simultaneously
  • Enables concurrent execution for performance and scalability
Key Benefits of Hyperledger Fabric Transaction Flow
  • Better reflects business processes by specifying who endorses transactions
  • Eliminates non-deterministic transactions
  • Scales the number of participants and transaction throughput

Q1. In a 51% attack, what does the attacker gain control of?
A More than 50% of all wallets on the network
B More than 50% of the network's total hashing power
C More than 50% of smart contracts
D More than 50% of private keys
A 51% attack requires gaining control of more than 50% of the network's total hashing power, giving the attacker decisive control over the blockchain.
Q2. What is a Sybil attack?
A An attack that modifies transaction hashes
B An attack that floods the network with transactions
C An attack using multiple fake identities assigned to the same node to surround victims
D An attack that steals private keys
A Sybil attack assigns multiple identities to the same node. The attacker controls many fake nodes, surrounding a victim node so all its transactions are intercepted, enabling double-spending.
Q3. What is a reentrancy attack in smart contracts?
A An external contract repeatedly calls a withdraw function before the state is updated
B An attacker mines blocks faster than the network
C A transaction is modified after being submitted
D A miner withholds a found block from the network
Reentrancy occurs when an external contract exploits a vulnerability to re-enter a function (usually withdraw) before the original contract updates its balance, allowing repeated fund draining.
Q4. What does SegWit (Segregated Witness) fix in Bitcoin?
A 51% attacks
B Double spending
C Private key theft
D Transaction malleability attacks
SegWit separates the signature from the TXID hash computation, making TXIDs non-malleable. Bitcoin adopted SegWit in 2017 specifically to fix transaction malleability attacks.
Q5. In the CIA Triad, which property does blockchain MOST strongly support?
A Confidentiality
B Integrity
C Authentication
D Accountability
Blockchain strongly supports Integrity. Its immutable ledger, cryptographic hashing (SHA-256), and distributed consensus ensure data cannot be tampered with. Note: Confidentiality is mixed (public chains are transparent).
Q6. In front-running attacks, what does the attacker exploit?
A Weak private keys
B Integer overflow vulnerabilities
C Publicly visible pending transactions in the mempool
D Weak consensus mechanisms
Front-running exploits the fact that transactions are publicly visible in the mempool before being mined. The attacker submits the same transaction with higher gas to be mined first.
Q7. What is selfish mining?
A Mining and withholding a found block to gain unfair advantage over honest miners
B Mining with stolen computing resources
C Refusing to validate other miners' blocks
D Creating fake transactions to clog the network
Selfish mining = finding a block, keeping it secret, continuing to mine privately. When honest miners find a block, the selfish miner releases their chain. If longer, it wins and honest miners' work is orphaned.
Q8. An eclipse attack achieves its goal by:
A Overwhelming a node with transactions
B Stealing the victim node's private key
C Modifying blockchain consensus rules
D Controlling all peer connections of a victim node, isolating it from honest nodes
An eclipse attack works by flooding a victim's peer table with malicious IPs so all connections go to attacker-controlled nodes. The victim is isolated and receives only the attacker's view of the blockchain.
Q9. In Hyperledger Fabric, which node type does NOT hold a ledger or chaincode?
A Endorsing Peer
B Ordering Node
C Committing Peer
D Fabric CA
The Ordering Node is responsible for ordering transactions into blocks and communicating with peers. It does NOT hold smart contracts (chaincode) or a ledger — that's the peers' job.
Q10. In Solidity ≥ 0.8.0, integer overflow/underflow is:
A Still possible and requires SafeMath
B Impossible due to fixed integers
C Automatically reverted by built-in overflow checks
D Handled by the EVM, not Solidity
Solidity ≥ 0.8.0 has built-in overflow/underflow checks that automatically revert the transaction when arithmetic exceeds bounds. SafeMath is only needed for older versions.
Q11. Long-range attacks target which type of blockchain?
A Proof of Stake (PoS) blockchains
B Proof of Work (PoW) blockchains
C Private/permissioned blockchains only
D All types of blockchains equally
Long-range attacks specifically target PoS blockchains. Attackers use old validator private keys to reconstruct a valid alternative history from genesis — something not possible in PoW (which requires actual energy/hash power).
Q12. What was the approximate amount stolen from cryptocurrency platforms in 2022?
A $500 million
B $1.7 billion
C $2.2 billion
D $3.7 billion
In 2022, approximately $3.7 billion was stolen from cryptocurrency platforms. This decreased to $1.7B in 2023, then rose again to $2.2B in 2024.
